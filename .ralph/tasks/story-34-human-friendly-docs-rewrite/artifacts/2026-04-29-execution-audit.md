## Authoritative facts to preserve

### Image publication

- CI builds four archives: `runner-image` and `verify-image`, each for `amd64` and `arm64`.
  - Source: `.github/workflows/publish-images.yml`
- GHCR publish script emits final refs as:
  - `ghcr.io/${ghcr_owner}/runner-image:${git_sha}`
  - `ghcr.io/${ghcr_owner}/verify-image:${git_sha}`
  - Source: `scripts/ci/publish-ghcr-multiarch-from-archives.sh`
- Quay is a mirror copied from GHCR and repository names are controlled by:
  - `RUNNER_IMAGE_REPOSITORY`
  - `VERIFY_IMAGE_REPOSITORY`
  - Source: `scripts/ci/publish-quay-from-ghcr.sh`

### Image-name contradiction to fix

- `README.md` currently tells operators to pull:
  - `ghcr.io/${GITHUB_OWNER}/cockroach-migrate-runner:${IMAGE_TAG}`
  - `ghcr.io/${GITHUB_OWNER}/cockroach-migrate-verify:${IMAGE_TAG}`
- That conflicts with CI and the operator guide, which both use `runner-image` and `verify-image`.
- Treat `runner-image` and `verify-image` as authoritative until proven otherwise.

### Runner public interface

- Binary usage:
  - `validate-config --config <PATH> [--deep]`
  - `run --config <PATH>`
  - global flag: `--log-format text|json`
  - Source: `crates/runner/src/lib.rs`
- Public HTTP endpoints:
  - `GET /healthz`
  - `GET /metrics`
  - `POST /ingest/{mapping_id}`
  - Source: `crates/runner/src/webhook_runtime/mod.rs`
- Webhook config:
  - `webhook.mode` supports `http` and `https`
  - default is `https`
  - `webhook.tls` is required for `https`
  - `webhook.tls` must not be set for `http`
  - `webhook.tls.client_ca_path` enables mTLS
  - Source: `crates/runner-config/src/config/parser.rs`
- Destination config:
  - either `destination.url`
  - or decomposed `host`, `port`, `database`, `user`, `password`, optional `tls`
  - mixed URL plus decomposed shape is invalid
  - decomposed TLS uses:
    - `mode`
    - `ca_cert_path`
    - `client_cert_path`
    - `client_key_path`
  - `verify-ca` and `verify-full` require `ca_cert_path`
  - client cert and key must be set together
  - Source: `crates/runner-config/src/config/parser.rs`

### Verify-service public interface

- CLI lives under the `verify-service` command with:
  - `validate-config --config <path>`
  - `run --config <path>`
  - per-command flag: `--log-format text|json`
  - Source: `cockroachdb_molt/molt/cmd/verifyservice/verifyservice.go`
- Listener config:
  - `listener.bind_addr` is required
  - `listener.tls.cert_path` and `listener.tls.key_path` must both be set when TLS is configured
  - `listener.tls.client_ca_path` enables mTLS
  - listener modes are `http`, `https`, `https+mtls`
  - Source: `cockroachdb_molt/molt/verifyservice/config.go`
- Database config:
  - both `verify.source.url` and `verify.destination.url` must use `postgres` or `postgresql`
  - if `sslmode` is `verify-ca` or `verify-full`, nested `tls.ca_cert_path` is required
  - nested client cert and key must be set together
  - `ConnectionString()` injects `sslrootcert`, `sslcert`, and `sslkey` from nested TLS config into the URL
  - Source: `cockroachdb_molt/molt/verifyservice/config.go`
- HTTP API:
  - `POST /jobs`
  - `GET /jobs/{job_id}`
  - `POST /jobs/{job_id}/stop`
  - `POST /tables/raw`
  - `GET /metrics`
  - Only one active job at a time.
  - Only the most recent completed job is retained.
  - Retained job state is lost on process restart.
  - Source: `openapi/verify-service.yaml`
  - Source: `cockroachdb_molt/molt/verifyservice/service.go`

### SQL setup generators

- Operator-managed SQL setup is generated by:
  - `scripts/generate-cockroach-setup-sql.sh`
  - `scripts/generate-postgres-grants-sql.sh`
- `generate-cockroach-setup-sql.sh` requires:
  - `cockroach.url`
  - `webhook.base_url`
  - `webhook.ca_cert_path`
  - `webhook.resolved`
  - `mappings[].source.database`
  - `mappings[].source.tables`
- `generate-postgres-grants-sql.sh` reads:
  - `mappings[].destination.database`
  - `mappings[].destination.runtime_role`
  - `mappings[].destination.tables`
- Source:
  - `scripts/generate-cockroach-setup-sql.sh`
  - `scripts/generate-postgres-grants-sql.sh`
  - `scripts/README.md`
