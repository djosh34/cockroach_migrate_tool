# Rust dependency report

Scope: Rust crates only, from `Cargo.toml` files and `Cargo.lock`. This report includes direct workspace dependencies and every transitive package present in the lockfile, including platform-specific and feature-conditional packages. No application code was changed.

## Executive summary

- Lockfile package entries analyzed: 257 total, including 4 local workspace crates and 253 external package entries.
- Direct runtime dependency groups: web/server (`axum`, `hyper-util`), async runtime (`tokio`), TLS (`rustls`, `rustls-pemfile`, `tokio-rustls`), database (`sqlx`), serialization/config (`serde`, `serde_json`, `serde_yaml`), errors (`thiserror`), and time/log formatting (`time`).
- Direct dev/test-only groups: HTTP client/proxy support (`reqwest`, `hyper`, `http-body-util`) and temporary filesystem isolation (`tempfile`). `serde` and `serde_yaml` are listed as runner dev-dependencies too, but they are also runtime dependencies through other workspace crates.
- Largest compile-time opportunities: `sqlx` for runtime builds, `reqwest` for test builds, then the in-process web/TLS stack. `tempfile` is an easy test-only candidate because a small internal temp directory helper could replace it.
- Dependencies that are probably not worth attacking directly: almost all transitive crates. They should be removed by replacing the direct parent that pulls them in, not by trying to depend on or patch them individually.

## Direct dependency recommendations

### `axum`

- Group: direct runtime dependency / Runtime HTTP routing/framework.
- Used for: Used by runner webhook runtime to define routes, handlers, extractors, status responses, and body handling. It sits on top of hyper/tower and is convenient but pulls a non-trivial web stack.
- Only for testing: No.
- How much used: Medium runtime usage: a few import sites, but central to serving the webhook and metrics endpoints. Direct source references found: 5 across 3 Rust files. Transitive subtree size: 52 packages excluding itself.
- Ease to remove: Medium: could remove by writing directly against hyper/hyper-util services, but that would make routing/extraction/error response code more manual.
- Simpler replacement / compile-speed option: Possible replacement: direct hyper service_fn or a tiny internal router. Fewer dependencies, more custom code. Keep unless compile time is a top priority.

### `http-body-util`

- Group: direct dev dependency; runtime transitive elsewhere / Dev/test HTTP body helper.
- Used for: Used by webhook chaos gateway test support to collect/build HTTP bodies.
- Only for testing: direct repository usage is test-only, but the crate is still present in the runtime graph through `axum`.
- How much used: Very low direct usage in one test support module. Direct source references found: 1 across 1 Rust files. Transitive subtree size: 6 packages excluding itself.
- Ease to remove: Easy: can be replaced with hyper body primitives or local helper code if test HTTP proxy is simplified.
- Simpler replacement / compile-speed option: Candidate for removal if tests can avoid the chaos gateway body conversion helper.

### `hyper`

- Group: direct dev dependency; runtime transitive elsewhere / Dev/test low-level HTTP.
- Used for: Direct dev dependency for webhook chaos gateway test support. Runtime gets hyper transitively through axum/hyper-util.
- Only for testing: direct repository usage is test-only, but the crate is still present in the runtime graph through `axum`/`hyper-util`.
- How much used: Very low direct usage. Direct source references found: 1 across 1 Rust files. Transitive subtree size: 29 packages excluding itself.
- Ease to remove: Easy to remove as direct dev dep only if chaos gateway is rewritten to reqwest/axum/test harness; not easy to remove from graph while axum remains.
- Simpler replacement / compile-speed option: Could replace direct test use with axum test server or reqwest-only harness, but runtime graph still includes hyper.

### `hyper-util`

- Group: direct runtime dependency / Runtime HTTP server glue.
- Used for: Used to serve accepted TCP/TLS connections with Tokio IO through hyper 1.x server utilities.
- Only for testing: No.
- How much used: Low direct usage, but it is core server plumbing. Direct source references found: 2 across 2 Rust files. Transitive subtree size: 44 packages excluding itself.
- Ease to remove: Medium: can be removed only if the server integration is rewritten or replaced by another server crate.
- Simpler replacement / compile-speed option: No obvious simpler crate with fewer deps if staying on hyper 1.x. If axum is removed too, a direct hyper stack may still need hyper-util.

### `reqwest`

- Group: direct dev/test-only dependency / Dev/test HTTP client.
- Used for: Used heavily by integration tests and support harnesses for HTTPS/mTLS and webhook calls.
- Only for testing: Yes.
- How much used: Heavy test usage. Direct source references found: 17 across 10 Rust files. Transitive subtree size: 127 packages excluding itself.
- Ease to remove: Medium: removing requires a replacement HTTP client in tests, especially TLS/mTLS support.
- Simpler replacement / compile-speed option: Potential replacement: hyper client or tiny custom client. But reqwest provides mature blocking + rustls TLS APIs. Compile-time win is large for test builds, but test code gets much more complex.

### `rustls`

- Group: direct runtime dependency / Runtime TLS primitives.
- Used for: Used for TLS server configuration, certificate/key loading types, and TLS errors.
- Only for testing: No.
- How much used: Moderate usage in webhook runtime and error types. Direct source references found: 12 across 3 Rust files. Transitive subtree size: 39 packages excluding itself.
- Ease to remove: Medium-hard: removable only if HTTPS/mTLS support is removed or delegated to a reverse proxy/process manager.
- Simpler replacement / compile-speed option: Simpler replacement is operational: require TLS termination outside the runner and keep the runner HTTP-only. Code replacement crates are not likely smaller.

### `rustls-pemfile`

- Simpler replacement / compile-speed option: Could replace with a small internal PEM parser only for the accepted key/cert shapes, but that is security-sensitive and not worth it unless TLS is removed.

### `sqlx`

- Group: direct runtime dependency / Runtime PostgreSQL access.
- Used for: Used for connect options, connections, transactions, dynamic queries, rows, and database errors throughout bootstrap, reconcile, webhook persistence, tracking state, and config validation.
- Only for testing: No.
- How much used: Heavy usage: many files and many references. Direct source references found: 51 across 11 Rust files. Transitive subtree size: 149 packages excluding itself.
- Ease to remove: Hard: it is the persistence layer. Removing it is a real database abstraction rewrite.
- Simpler replacement / compile-speed option: Possible replacement: tokio-postgres/postgres-protocol stack may be lighter than sqlx, but would require manual row extraction, pooling/connection handling choices, and query error mapping. Biggest compile-time opportunity but also largest implementation cost.

### `tempfile`

- Group: direct dev/test-only dependency / Dev/test temporary filesystem.
- Used for: Used heavily by integration tests and test support to isolate config files, Postgres data dirs, cert outputs, and runner state.
- Only for testing: Yes.
- How much used: Heavy test usage. Direct source references found: 97 across 8 Rust files. Transitive subtree size: 15 packages excluding itself.
- Ease to remove: Easy-medium: std::env::temp_dir plus custom unique directories and cleanup could replace it.
- Simpler replacement / compile-speed option: A tiny internal tempdir helper would remove tempfile and its OS/random deps. This is one of the easiest compile-time wins for tests.

### `thiserror`

- Group: direct runtime dependency / Runtime error derivation.
- Used for: Used to derive Error implementations for runner and runner-config error enums.
- Only for testing: No.
- How much used: Low direct usage but valuable for many error variants. Direct source references found: 2 across 2 Rust files. Transitive subtree size: 5 packages excluding itself.
- Ease to remove: Easy-medium: could hand-write std::error::Error and Display implementations.
- Simpler replacement / compile-speed option: Replacement is manual impls. That removes proc-macro deps from thiserror but increases boilerplate and risks worse errors.

### `time`

- Group: direct runtime dependency / Runtime timestamps/durations formatting.
- Used for: Used by operator-log timestamps, runtime plans, metrics/test time handling, and reconcile timing.
- Only for testing: No.
- How much used: Moderate usage. Some usage is std::time, but the crate is needed for formatted timestamps. Direct source references found: 28 across 19 Rust files. Transitive subtree size: 12 packages excluding itself.
- Ease to remove: Medium: can be reduced if operator logs use SystemTime formatting or chrono-like alternatives; removing entirely requires replacing formatting.
- Simpler replacement / compile-speed option: No smaller obvious replacement for robust RFC3339-ish formatting. Could hand-roll narrowly for UTC timestamps if desired.

### `tokio`

- Group: direct runtime dependency / Runtime async executor/network/time.
- Used for: Used for async main, TCP listener, timers, spawning concurrent HTTP/reconcile work, and async DB/network integration.
- Only for testing: No.
- How much used: Moderate direct usage, foundational for sqlx/hyper/axum. Direct source references found: 12 across 6 Rust files. Transitive subtree size: 13 packages excluding itself.
- Ease to remove: Hard: async stack depends on Tokio. Removing means major architecture change.
- Simpler replacement / compile-speed option: No simpler replacement while using sqlx/hyper/axum. Keep.

### `tokio-rustls`

- Group: direct runtime dependency / Runtime Tokio TLS adapter.
- Used for: Used to accept TLS connections on Tokio sockets.
- Only for testing: No.
- How much used: Low direct usage but necessary for in-process HTTPS/mTLS. Direct source references found: 2 across 2 Rust files. Transitive subtree size: 46 packages excluding itself.
- Ease to remove: Medium: remove with in-process TLS or replace server TLS integration.
- Simpler replacement / compile-speed option: Operational replacement is external TLS termination. Code replacement is unlikely to reduce deps meaningfully.

## Suggested removal order

1. Replace `tempfile` in tests with a tiny local temp directory helper if test compile time matters. It is dev-only, heavily used, and not conceptually hard to replace.
2. Consider replacing `serde_yaml` by moving config files to JSON or TOML. JSON would reuse the existing `serde_json`; TOML would add a different parser and needs a separate dependency review.
3. Consider whether in-process TLS is required. If TLS can be terminated outside the runner, `rustls`, `rustls-pemfile`, and `tokio-rustls` become removable direct runtime dependencies and a large TLS subtree disappears.
4. Consider replacing `reqwest` in integration tests only if test build time is a serious pain. It is large, but it buys very convenient blocking HTTPS/mTLS client behavior.
5. Treat `sqlx` as the largest runtime compile-time lever, but do not remove it casually. Replacing it with `tokio-postgres` or a lower-level protocol stack is a full persistence-layer rewrite.
6. Leave `tokio`, `serde`, and `serde_json` in place unless the architecture changes dramatically. They are foundational and already standard choices.

## Notes and limitations

- I did not run `cargo`; analysis was done from manifests, lockfile parsing, and source searches in accordance with the repository instruction to use Nix rather than running Cargo directly.
- Source reference counts are lexical counts of crate import tokens, so derive macros and transitive usage can be undercounted. They are useful for relative sizing, not exact semantic reachability.
- Cargo.lock does not preserve a perfect runtime-vs-dev edge label for each transitive package. I separated direct runtime roots from direct dev roots and classified transitive packages by reachability from those roots.
